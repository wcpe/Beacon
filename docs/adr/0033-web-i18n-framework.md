# ADR-0033：管理台引入 react-i18next 国际化框架（zh-CN 先行、文案 key 化）

**状态**：已接受

## 背景

管理台（`web/`）的全部可见文案此前都**硬编码在组件里**：导航名、页标题、按钮、表头、占位符、空态、提示、状态 / 角色显示名（如 `BC 代理` / `子服`、`生效` / `已过期`）等中文字符串散落在各 `.tsx` 中。审计页的 `action` 是后端英文枚举（`config.publish` / `namespace.update` 等，见 `internal/model/enums.go`），前端直接原样展示英文，运维看不懂。

后果：

1. **文案无单一真源**：同义文案（"取消" / "查询" / "操作" / "环境"）在多页重复，改一处词要全局找替换，易漏。
2. **无法支撑多语言**：现状把语言写死进 DOM，未来加英文 / 其他语言只能逐字符串重写。
3. **审计动作不可读**：英文枚举直接落到表格，运维需自行翻译。

FR-50 要求引入完整 i18n 框架、把全站硬编码中文 key 化、并把审计 action 经 i18n 映射显示中文。这是一次**横切的前端重构**，须先定下框架选型、迁移原则与 key 组织，避免边做边改造成风格漂移。

## 决策

1. **采用 `react-i18next` + `i18next` 作为前端 i18n 框架。** 它是 React 生态事实标准：`useTranslation()` Hook + `t('key')` 取值、命名空间 / 嵌套 key、插值与复数齐备，与现有 React 18 + Vite + TS 栈零摩擦，无需额外构建插件。本期**只交付 zh-CN 全量**；框架就绪后加 en 只需补一份资源文件，不再改组件——这正是引入框架（而非简单常量表）的目的。

2. **i18n 同步初始化、资源内联、`lng`/`fallbackLng` 均为 `zh-CN`。** 在 `web/src/i18n/index.ts` 用 `i18n.use(initReactI18next).init({ resources, lng: 'zh-CN', fallbackLng: 'zh-CN' })` 同步初始化，资源对象（`locales/zh-CN.ts`）直接 import 内联、不走异步 HTTP 后端加载——保证**测试环境（vitest jsdom）里 `t()` 同步返回真实 zh-CN 文案、永不出现裸 key**。生产入口 `main.tsx` 与测试 `test/setup.ts` 各 import 一次该模块完成接线。

3. **zh-CN 翻译值逐字等于现有中文文案——等值迁移、不改词。** 本次是"搬运文案、不是改文案"：每个 key 的 zh-CN 值 = 原硬编码字符串原文。由此**渲染出的可见文本完全不变**，既有断言中文文本的 vitest 用例绝大多数无需改动即继续通过，把回归面压到最小。仅当某测试断言的 DOM 结构 / 取值方式本身变了才动它，且只做最小修。

4. **状态 / 角色 label 一并 key 化，但保留当前可见文本。** 角色（`bungee→BC 代理` / `bukkit→子服`）原本就显示中文 → key 化为中文值；健康状态（`online` / `lost` / `offline`）与密钥状态等**原本就显示英文枚举原文**的 → key 化后 zh-CN 值仍为该英文原文（如 `status.online` → `online`），维持渲染不变、不擅自改成"在线 / 失联 / 离线"（那会改变可见文本、破坏既有测试，且属本期范围外的文案设计变更）。

5. **审计 action 经 i18n key 映射显示中文。** 后端**不改、仍返英文枚举**；前端建 `audit.action.<枚举>` 键族（如 `audit.action.config.publish` → `发布配置`），渲染时 `t('audit.action.' + action, { defaultValue: action })`——已知枚举显示中文、未知枚举回退原文，不出现裸 key。

6. **key 命名分层、稳定、同义归并到 `common`。** 顶层按域分组：`nav.*`（导航）、`common.*`（跨页复用：查询 / 取消 / 操作 / 删除 / 环境 / 大区 / 小区…）、`audit.*` / `instances.*` / `configs.*` / `zones.*` 等各页域、`status.*` / `role.*`（label）。同义文案合并进 `common` 避免重复（DRY）；key 用小写点分层、语义命名，不用序号。

## 理由

- **react-i18next 是最省的达标路径**：满足"完整框架 + 可加语言"，无新构建链、社区成熟、与 React 栈天然契合；自造常量表无法满足"框架"要求且复用 / 插值 / 语言切换都要重造轮子。
- **同步内联初始化**直接消除"测试里异步加载拿不到文案 → 裸 key"这一最大失败模式，让 `t()` 在 jsdom 下行为与生产一致。
- **等值迁移原则**把一次横切大改的回归风险降到最低：可见文本不变 ⇒ 测试基本不动 ⇒ 改动可信。
- **状态英文保值 + 角色中文**忠实保留现状语义，不在 i18n 接入里夹带文案设计变更（范围纪律）。
- **审计 action 映射 + defaultValue 回退**让运维读懂动作，又对后端新增枚举鲁棒（不崩、不裸 key）。

## 后果

- 新增 `web/src/i18n/` 目录（`index.ts` + `locales/zh-CN.ts`），新增依赖 `react-i18next` / `i18next`（FR-50 已授权）。
- 各组件从硬编码中文改为 `const { t } = useTranslation()` + `t('key')`；`main.tsx` 与 `test/setup.ts` 各加一行 i18n import。
- 后续加语言只需补 `locales/<lng>.ts` 并在 `resources` 注册，组件零改动。
- zh-CN 资源文件成为**前端文案单一真源**；新增 / 改文案改这里，不再散落组件（与 doc-sync 的"单一真源"一致）。
- 翻译键覆盖不全会回退到 `fallbackLng`（仍 zh-CN）或 `defaultValue`，不会显示裸 key；但漏 key 会导致"该处不可译" —— 以"渲染文本不变"为验收的回归测试可挡住绝大多数遗漏。

## 备选方案

- **自造文案常量表（`const T = { ... }`）+ 手写取值**：不引第三方依赖，但不满足 FR-50"完整框架"要求，且语言切换 / 插值 / 复数 / 回退全要自己实现，等于重造一个残缺 i18next。被否。
- **`react-intl`（FormatJS）**：能力相当，但 ICU message 语法更重、与现有简洁文案不匹配，且需 `<FormattedMessage>` 组件式用法侵入更深；react-i18next 的 `t()` 函数式更贴近现有渲染写法。被否。
- **本期直接把状态改成中文（在线 / 失联 / 离线）**：会改变可见文本、破坏既有 `StatusBadge` 等断言英文的测试，且属"文案设计变更"而非"i18n 接入"，越出 FR-50 范围。被否（保留英文原值，仅 key 化）。
- **异步 HTTP 后端加载语言包（i18next-http-backend）**：生产可懒加载，但测试环境拿不到同步文案 → 裸 key，且本期只有 zh-CN、产物经 `go:embed` 内嵌单二进制，懒加载无收益。被否（资源内联同步初始化）。
