# ADR-0016：agent 内置跨服消息中间件，基于 Redis 的通用通信层

**状态**：已接受

## 背景

BC 与 Bukkit 子服原版通信走 PluginMessage：广播式（任何服都收到）、需在线玩家当载体（空服发不出去）、有大小限制、**没有可靠回调与稳定返回**。业务插件要做跨服请求-响应只能各自硬凑，脆弱且重复造轮子。

Beacon 已提供注册/发现/健康/配置（控制面），但**明确不传游戏数据**（[architecture-invariants](../../.claude/rules/architecture-invariants.md) #1、[PRD](../PRD.md) §1 Non-Goals）。现需要一个**通用、可靠、可定向、能等回信**的服务器间传输层，供③层业务插件（匹配/对战/存储/排行等）复用，同时**不把游戏逻辑塞进控制面、不把控制面拽进玩家热路径**。

设计空间评估了 Actor 框架（Akka/Orleans/Skynet）、成品后端（Nakama）、编排平台（Agones）、消息队列（Kafka/RabbitMQ）。结论：**借鉴其思想，不引入其重型实现**（见「备选方案」）。

## 决策

1. **分三层、各司其职，本 ADR 只立②层**：①控制面 Beacon（事实/配置/发现/调度决策）；②通用通信中间件（本 ADR）；③业务功能插件（匹配/对战/存储/排行，独立项目，**不在本 ADR 范围**）。②对③只提供与内容无关的传输，不理解任何业务语义。

2. **中间件是数据面，不经过 Beacon 控制面**：消息只在 `agent ↔ Redis ↔ agent` 之间流动。Beacon 仅提供"谁在哪"的地址簿；Beacon 不可用时，中间件用缓存地址继续工作。**控制面永不在消息路径上。**

3. **传输用 Redis；可靠骨干用 Redis Streams**：
   - **可靠送达**（定向发送、按玩家寻址、需补偿的事件）走 **Redis Streams + 每服消费组**：订阅者离线期间消息留存，重新上线从上次位置补消费（至少送达一次 + 业务侧幂等）。
   - **可丢事件**（不在乎丢失的广播）可走 **pub/sub**。
   - RPC 回信走轻量回信通道（两端在线 + 超时兜底，不持久化）。

4. **四种消息模式**：定向发送（fire-and-forget）、**请求-响应（RPC，带关联ID + 超时）**、主题发布订阅、按玩家所在服寻址。

5. **按玩家寻址依赖玩家位置名册**：由 **BC 上的 beacon-proxy 维护**"玩家→所在子服"索引（存 Redis），随玩家进服/换服/退出更新；发"给玩家X"时先解析其所在服，再走定向发送。
   - **一致性取舍（简单优先）**：接受换服瞬间的短暂错位，不做强一致；解析落空（玩家已不在该服）走"找不到目标"兜底（重试一次或丢弃），不为极小概率上分布式一致性。
   - **proxy 重启重建**：不信任旧名册，重启后**重新扫描当前在线玩家重建**索引。

6. **落点：agent 内独立可开关模块**：住 agent 代码库，但**单独模块 + 配置开关**（默认关），与配置同步/心跳**两套连接、两套线程，故障域隔离**——Redis 挂不连累配置命脉。

7. **沿用传输抽象（遵循 [ADR-0005](0005-agent-transport-codec-abstraction.md)）**：core 只依赖 `MessageTransport` / 既有 `JsonCodec` 抽象，**Redis 客户端关在适配器里**，core 不硬绑 Jedis/Lettuce；以后可换 NATS 等而不动业务层。

8. **不在 MC 主线程做阻塞 IO**：订阅在独立线程收，默认在异步线程把消息交给业务插件；handler 需碰 Bukkit/Bungee API 时自行切回平台线程（中间件提供帮助方法）。

9. **fail-static 不破坏**：中间件不可用时，`send` 报失败、RPC 超时、订阅暂停；但配置同步照常、**玩家照常进服游玩**（玩家连接是 BC/Bukkit 原生）。

10. **Docker 兼容**：Beacon（单二进制 + DB）与 Redis 作为 compose 服务即可。**关键约束**：agent 上报的 `address` 必须是同网络下可达地址（容器服务名/可路由 IP，非 localhost），否则目录注入与 agent 互连失败。

11. **单 BC 入口为明确前提**：集群只有一个流量入口（类比 nginx 单入口），玩家位置名册与 Streams 消费组命名按单 proxy 设计。多 BC 不在本 ADR 范围；未来如需多入口，另立 ADR 处理名册归属与命名冲突。

12. **Streams 必须设裁剪上限**：每条流设 `MAXLEN`/时间窗口上限，旧消息自动淘汰，防止无限增长吃光内存；离线超过窗口的子服上线即视为"错过不补"。具体阈值实现时按 Redis 内存定。

13. **消息信封带类型 + 版本号**：每条消息含 `type` 与 `version`；演进规约"**只增不改**"——可加可选新字段，不删除/不重命名/不改语义已有字段，保证集群内新老插件混跑向后兼容。

14. **Redis 客户端自带，绝不经 CoreLib**：经 CoreLib 取 Redis 会违反不变量 #5（agent 不依赖 CoreLib），且与 [ADR-0014](0014-downstream-identity-source-direction.md)（CoreLib 依赖 agent 取身份）构成循环依赖。agent 自带 **Jedis**，经 TabooLib `@RuntimeDependencies` 运行期下载、不 shade，藏在 `MessageTransport` 适配器后（core 不 import Jedis，简单优先、可换 Lettuce/NATS 不动业务层）。

15. **Redis 连接配置由 Beacon 配置中心下发**：host/port/db 经有效配置下发到各 agent，全集群同一份、可热更可回滚（FR-3）；**Beacon 自身从不连接 Redis**，只下发配置文本，故不违反 [ADR-0003](0003-no-redis-in-mvp.md)。**agent 无 env，密码也只能经 Beacon 下发**，而 Beacon 配置库在 FR-20 配置加密前为明文——故**带密码鉴权的 Redis 下发依赖 FR-20 先行（FR-20 由 P3 提前为 FR-26 前置）**。运行机制：配置变更触发消息模块重连；Redis 配置入本地快照供 fail-static；冷启动未取得配置时消息模块先关。

## 理由

- agent 本就在数据面（寄生于游戏服进程），由它提供传输**不会**把控制面拽进热路径——这与"别让 Beacon 当电话线"不矛盾。
- Redis Streams 在本规模（约 50 服、每秒数百~数千条）已完整覆盖 MQ 所需能力（持久化/消费组/补偿/重放），且本就要引 Redis，**无需再加 Kafka/MQ**。
- 通用传输与业务功能分离，避免把 agent 变成"配置+发现+消息+业务"的上帝插件，保住故障隔离与简单优先。
- 复用 Beacon 发现做服解析、复用 ADR-0005 传输抽象，最小化新增面。
- **不与 [ADR-0003](0003-no-redis-in-mvp.md) 冲突**：ADR-0003 约束的是「MVP（P1）**控制面**不引 Redis」；本 Redis 在**数据面**（agent 侧业务消息）、属 **P3**、独立可关、**不进 Beacon 控制面进程**，是该约束之外的另一范畴，非推翻。

## 后果

- 新增一个数据面依赖 **Redis**：它是"消息这块"的单点，挂了消息不可用，但**不影响配置同步与玩家游玩**。需要更稳时后续可上 Redis 哨兵/集群（非本轮）。
- beacon-proxy 多一项职责：维护玩家位置名册（额外工作量与一致性考量）。
- 这是对 [PRD](../PRD.md) §1 Non-Goals「跨服通信由业务插件自实现」的**有意识边界澄清**：提供传输管道≠实现游戏功能；功能仍归③层业务插件（PRD 该条同步澄清）。
- 与 FR-13（完整虚拟合区运行时通道 / 控制面 HA）的关系：本 ADR 只前移"通用传输管道"这一最小基础设施，**不关闭 FR-13**。
- 业务功能（匹配/对战/存储/排行）属③层、独立立项，**不得**借本 ADR 之名塞进 Beacon 或中间件。
- **FR-26 带鉴权 Redis 依赖 FR-20 配置加密先行**：FR-20 由 P3 提前为 FR-26 前置（顺序 FR-20 → FR-26 密码下发）。未加密前若需先验证，可临时用无密码内网 Redis。

## 备选方案

- **Kafka / RabbitMQ**：为百万级数据管道/复杂路由设计，远超本场景；额外重型运维件，撞简单优先。**否决**。需海量事件留存做分析时，再作③层独立项目评估。
- **Akka / Pekko（JVM Actor 集群）**：与 Bukkit 插件线程/类加载严重不合，Akka 还是 BSL 商业许可；只借其"逻辑ID寻址 + 位置透明"思想，**不引入框架**。
- **Nakama（成品后端）**：客户端直连的权威后端，与"现有 BC+Bukkit 服间总线"是两套架构，且与 Beacon 发现/配置重复；只借其"能力拆分 + 类型化 RPC"思路，**不运行其服务**。
- **Agones（k8s 编排）**：需整集群上 k8s，且与 Beacon 注册/健康/FR-10 重叠；只借其"服务器生命周期状态机"喂给未来 FR-10，**本轮不引入**。
- **仅 pub/sub（不持久化）**：实现最轻，但订阅者离线即丢消息，不满足"离线也能补"的要求；故选 Streams 为可靠骨干。
- **消息走 Beacon 控制面中转**：会把控制面拽进热路径、把游戏逻辑引入控制面，撞架构红线。**否决**。
