# ADR-0017：流量调度——控制面给决策、数据面做执行；drain 落 DB、canary 划范围外

**状态**：已接受

## 背景

第二期（P2）要做流量调度（落位均衡 / drain）。一个 zone 下常有多台同质子服，需要统一依据回答"新玩家落哪台"，以及标记"哪台在维护、暂时别再送人（drain）"。这里有三处必须先拍板的边界，否则极易把游戏逻辑漏进控制面、或被 agent 上报的易变指标污染决策：

1. 调度是控制面**执行**（直接路由 / 踢人）还是只**给建议**？
2. drain 这种运维状态位是放内存运行态还是落 DB？
3. canary 引流（按比例灰度放量）本期做不做？

## 决策

1. **控制面只给调度决策（query-only），玩家连接的实际落位由数据面执行。** 控制面提供"某 zone 内推荐落位子服"的排序建议与 drain 状态查询；它不发起、不路由、不中断任何玩家连接。这守住架构红线"控制面不写游戏逻辑、不碰玩家连接"。

2. **落位排序只用权威事实，不用 agent 上报的易变指标。** 排序依据 = `status=online` ∧ 未 drain，再按 weight 降序 → capacity 降序 → serverId 升序（确定性、可穷举测）。**不读 `playerCount` / `tps`**——二者在注册表里既定为"仅展示、不参与决策"，把它们喂进决策会破坏该不变量、且让控制面越界感知活跃负载。活跃负载的精细落位由掌握实时连接的数据面负责。

3. **drain 是运维决策，落 DB（与 `zone_assignment` 同源类别），不放内存运行态。** drain 由管理员设置、须跨控制面重启存活、要可审计，属"事实/决策"而非"注册/健康运行态"。故新增 `server_drain` 表（GORM AutoMigrate、零方言绑定、软删取消），读落位时把 drain 集合叠加到内存实例视图上剔除候选。注册/健康仍以进程内存为真源，两者不互为权威。

4. **canary 引流划在本期范围外。** 按比例灰度放量 / 金丝雀引流是更细的 P2/P3 能力，本期不做、不引入 `canary` 字段。确需要做时，先写新 ADR 取代 / 补充本条，界定"比例从哪来、谁执行引流"，再动代码——不在落位接口里静默塞 canary。

## 理由

- 落位/排空"建议 vs 执行"分离，让控制面继续只存事实 + 给查询，数据面（如 Bungee proxy）按建议落位、按 drain 停止送人，职责清晰、不触碰 MVP 已锁定的控制/数据面边界。
- 决策只吃权威事实（weight/capacity/online/drain），结果确定、可单测、可审计；不被 agent 上报抖动影响。
- drain 落 DB 与 zone 指派同构，复用既有软删 / 事务 + 审计模式，代价低、可移植。
- 明确把 canary 排除，避免范围漂移与"为未来预留空壳"。

## 后果

- 控制面给的是"按事实排序的候选"，不是"按实时人数精排的最优解"；最终落位与负载精排由数据面承担。这是边界使然，不是缺陷。
- drain 读路径需在内存实例视图上叠加一次 DB drain 集合（按 ns 一次性取，非每实例查询，无 N+1）。
- 后续若要实时负载精排 / canary，须新 ADR 先行。

## 备选方案

- **控制面直接执行落位 / drain 时强制踢人**：把玩家连接逻辑拉进控制面，违反控制/数据面边界红线。被否。
- **落位读 agent 上报的 playerCount 实时精排**：破坏"playerCount 仅展示、不参与决策"不变量，且让控制面越界感知活跃负载。被否（精排归数据面）。
- **drain 放内存运行态**：控制面重启即丢失维护标记，运维不可接受。被否。
- **本期顺带做 canary**：超出 P2 当前切片、易镀金。被否，留新 ADR。
