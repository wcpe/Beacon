# ADR-0003：第一期不引入 Redis

**状态**：已接受

## 背景

旧方案（ZoneManager）用 Redis 作跨服总线，因为当时没有中心服务。Beacon 有了中心控制面后，需重新评估是否还需要 Redis。

## 决策

**MVP 不引入 Redis**（也不引 MQ）。服务注册表与健康状态放 Go 进程内存（`map + RWMutex`）；配置/版本/分配/审计落 MySQL 为权威；动态配置推送用 **REST 长轮询**（见 [ADR-0006](0006-rest-long-poll-push.md)）。

## 理由

- 控制面单节点，注册/健康完全可放进程内存（50 服规模毫无压力）。
- 配置推送用长轮询（Nacos 本尊即如此），不需要 Redis pub/sub。
- 少一个有状态中间件 = 少一份部署/运维/故障面，docker-compose 仅 `beacon + mysql` 两容器，更干净（简单优先）。

## 后果

- 控制面重启会丢挂起的长轮询，但 agent 断连后指数退避重连即恢复，无需持久化挂起态。
- **Redis 仅在 P3 控制面做 HA（多节点）时才回来**，届时作为跨节点的 pub/sub + 共享态。当前不预留空壳。

## 备选方案

- **Redis pub/sub 推送 + 共享注册表**：为单节点引入分布式中间件，过度工程。被否（推迟到 HA 阶段）。
