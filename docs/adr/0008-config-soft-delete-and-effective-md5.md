# ADR-0008：配置软删唯一性与有效配置 md5 的实现取舍

**状态**：已接受

## 背景

第一期设计经对抗式评审后，发现两处"看似正常实则有坑"的实现细节，需明确记录为什么这么做，避免后人误改回踩坑的写法。

## 决策

### 1. 配置项软删唯一键用固定哨兵默认值
`config_item` 的唯一键含 `deleted_at`，且 `deleted_at` 默认值取**固定哨兵** `1970-01-01 00:00:00`（非 NULL）、软删时填真实删除时间。

理由：若 `deleted_at` 默认 NULL，则 MySQL 与 Postgres 的唯一索引都**不把多个 NULL 视为冲突**，于是"未删的同标识配置项"在数据库层根本拦不住——唯一键形同虚设。用固定哨兵值纳入唯一键：未删记录哨兵值相同 → 唯一约束真正生效；软删后值变真实时间 → 允许同标识重建。两方言行为一致。

### 2. 有效配置整体 md5 把 dataId 名纳入哈希
整体 md5 = `md5(concat(dataId + ":" + 单dataId_md5))`，而非纯拼接各 dataId 的 md5。

理由：纯拼 md5 时，配置集合 `{A: md5x}` 与 `{B: md5x}` 会算出相同整体 md5，agent 长轮询会误判"没变"。把 dataId 名纳入哈希消除这种集合碰撞。

## 后果

- 软删唯一约束真实生效、跨 MySQL/Postgres 一致，不依赖方言专有的部分索引。
- 有效配置变更判定不会被 dataId 集合碰撞骗过，长轮询热更可靠。

## 备选方案

- **软删用部分索引 `WHERE deleted_at IS NULL`**：Postgres 支持，MySQL 8.0.13+ 函数索引写法不同 —— 引入方言分叉，违反 GORM 可移植（[ADR-0002](0002-go-react-embedded-stack.md) 与架构不变量）。否决。
- **整体 md5 纯拼接子 md5**：实现简单但存在集合碰撞，已被对抗式评审证伪。否决。
