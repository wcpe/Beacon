# ADR-0019：健康告警通道做成可扩展抽象（站内信 + webhook）

**状态**：已接受（「告警不落库」一条被 [ADR-0041](0041-alert-event-persistence.md) 取代——FR-89 新增 `alert_event` 留痕表作只读历史信息流；通道仍不持真源、健康真源仍在内存，本 ADR 的告警通道抽象不变）

## 背景

FR-28（增强 FR-5）要求在实例失联/状态异常时**主动告警**，并明确"告警通道做成可扩展抽象，第一版实现站内信 + webhook，后续新通道只需实现接口即可接入"。

控制面已有一个后台健康扫描 goroutine（`HealthScanner`）按 TTL 推进状态机。告警的触发点天然在它每轮拿到的"状态变更集合"上。问题是**如何组织多种告警出口**，既满足第一版的站内信 + webhook，又不让以后加邮件/IM 通道时去改扫描器或既有通道。

## 决策

告警出口抽象为单一职责接口，扫描器只依赖抽象、经分发器扇出：

```go
// 一个告警通道：把一条告警投递出去
type Alerter interface {
    Name() string
    Notify(ctx context.Context, a Alert) error
}

// 分发器：持有若干通道，逐个扇出、逐通道兜错，不阻断扫描
type Dispatcher struct { channels []Alerter }
func (d *Dispatcher) Dispatch(ctx context.Context, a Alert)
```

第一版两个实现：

- `InboxAlerter`（站内信）：进程内固定容量环形缓存 + **独立** `RWMutex`，`List()` 供管理台只读（`GET /admin/v1/alerts`）。不落库（与"注册/健康真源在内存"一致，告警是健康事件的派生，不持久化）。
- `WebhookAlerter`：HTTP POST 告警 JSON 到运维预设 URL，带超时。

告警**仅在实例进入异常态（degraded/lost/offline）时触发**；恢复到 online 不告警（避免噪音）。分发**逐通道兜错**：某通道失败仅 WARN 日志、继续下一个，绝不阻断健康扫描。

## 理由

- 端口适配器 / 依赖倒置：扫描器与既有通道对"新增通道"封闭、对扩展开放（OCP）——加通道 = 新增一个 `Alerter` 实现并注入分发器，扫描器与站内信/webhook 零改动，杜绝 `switch channelType` 式分支堆砌。
- 单一职责：`Alerter` 只管"投递一条告警"，`Dispatcher` 只管"扇出 + 兜错"，扫描器只管"判定状态变更 → 触发"。
- 守架构不变量与质量底线：站内信锁与三大运行态锁（Registry/Hub/Health）**不嵌套**；webhook 的 HTTP IO 在任何注册表锁之外（扫描器循环里调用）；告警不入库，不引中间件（无 Redis/MQ）。

## 后果

- 多一层接口 + 分发器（极薄），换来通道可扩展与可测性（测试注入假通道断言扇出/兜错），值得。
- 告警是进程内派生事件、不持久化：控制面重启后站内信清零。这与"健康真源在内存、重启重建"一致，可接受。
- 本期不做告警去重/静默窗口/持久化；需要时按通道或在分发器前置一层收敛，不改本抽象。

## 备选方案

- **在 `HealthScanner` 里直接写站内信 + 调 webhook**：少一层，但加通道要改扫描器、违反 OCP，且把 HTTP IO 焊进扫描器。被否。
- **告警落 MySQL 表 + 管理台查库**：可跨重启留存，但健康/注册真源本就在内存、告警是其派生，落库引入与真源切分相悖的第二权威，且 MVP 不为低价值持久化加表。被否（如需留存另写 ADR）。
- **引入消息队列做告警总线**：违反"MVP 禁重型件"（ADR-0003 同源精神），被否。
