# ADR-0036：区分配改派安全——排空门后端硬闸 + 高摩擦显式改派

**状态**：已接受

## 背景

zone（区）分配的权威由控制面 DB `zone_assignment` 表掌握（[ADR-0004](0004-zone-authority-control-plane.md)：serverId 由 agent 上报、zone 归属由控制面 DB 权威指派、agent 不声明 zone）。FR-35（[ADR 无]，已交付@v0.6.0）给它加了一个**看板式归派台**：未指派池 + 按大区分桶，**拖拽卡片即指派 / 改派 / 取消指派**（@dnd-kit），复用 `PUT/DELETE /admin/v1/zones/assignments`、后端零改动。

拖拽式归派好用，但对一个**高危、低频、强一致**的拓扑操作而言**太轻**：

1. **改派会热更该服的有效配置**。agent 拿 `(namespace, serverId)` 经 `zone_assignment` 解析出 `(group, zone)`，再按 global/大区/小区/单服四层 scope 覆盖链深合并出有效配置并热重载（见 ARCHITECTURE §5/§6）。把一台**在场有玩家**的服从区 A 改到区 B，等于在玩家头上换了一整套小区/大区层配置并触发热重载——轻则配置抖动，重则扰动正在进行的游戏。
2. **拖拽即改、无任何确认**：手一滑把卡片拖错桶就立即落库改派，没有"再想一下"的摩擦；这类"删库级"误操作本该高摩擦。
3. **前端摩擦不是安全边界**：拖拽确认即便加上，前端始终可被绕过（直接打 `PUT /zones/assignments`）。真正的安全必须在后端纵深兜底。

FR-71「区分配安全化」即在**不推翻 ADR-0004 的 DB 权威指派、不改 agent 行为、不动 `zone_assignment` 表结构与既有唤醒机制**的前提下，给改派加一道**后端硬闸（排空门）** + 一层**前端高摩擦显式操作**，并把面向用户的「zone」改称「区」。本 ADR 在 ADR-0004 基础上**扩展**改派的安全前置，不取代它。

## 决策

1. **后端排空门：在线非空的服，任何会改变其区归属解析的操作一律硬拒（409）。** 在 `ZoneService` 的指派 / 取消指派路径加一道纵深闸：若目标 serverId 在内存注册表中 **`Status == online` 且 `PlayerCount > 0`**（在场有玩家），且本次操作**会改变它解析出的 `(group, zone)`**——含**改派**（已指派 → 改到不同区）、**取消指派**（已指派 → 取消，归属变 global 兜底）、**首次指派**（未指派 → 赋区）——则返回专门错误码 `ZONE_SERVER_ONLINE_NONEMPTY`（HTTP 409），不落库、不审计、不唤醒。**运维须先排空**（drain，FR-10/[ADR-0017](0017-traffic-scheduling-decision-vs-execution.md)，或等玩家自然离开）使 `PlayerCount == 0` 后再操作。判据复用既有内存注册表的 `Status` 与 agent 上报的 `PlayerCount`，**不新增数据 / 不查库 / 不碰 agent**。

2. **同值 no-op：指派到当前完全相同的 `(group, zone)` 不落库、不审计、幂等返回成功。** 改派前先比对目标与现有 `zone_assignment`：完全相同即"什么都没变"，直接返回现有记录，**不写库、不记审计、不触发唤醒**。这既避免冗余审计噪声与无谓的覆盖链重算唤醒，也让排空门只拦"真正的变更"——同值即便在线非空也放行（因为它不改变任何东西）。

3. **取消"拖拽即改"，改为高摩擦显式改派 + 手输 serverId 二次确认 + 看板默认只读。** 前端移除 @dnd-kit 拖拽归派（draggable/droppable/onDragEnd 一并删除）；归派看板**默认只读**，须显式打开「解锁改派」开关后才出现「改派」/「取消指派」入口；改派走一个确认对话框，要求**手输目标 serverId 原样复述**方可提交（像 GitHub 删库要手打仓库名），**空服 / 离线服也要求此确认**（前端摩擦防的是"手滑"，与服是否在场无关）。纯前端 UX 摩擦，**REST 契约不变**（仍 `PUT/DELETE /admin/v1/zones/assignments`）。

4. **后端是唯一安全权威，前端摩擦只是第一道 UX 防线。** 排空门与同值判定一律在后端 `ZoneService` 裁决并落审计（既有 `zone.assign` / `zone.move` / `zone.unassign` 审计保留，真正发生的变更才记）。前端的"解锁 + 手输确认"可被绕过、只防误触；**在场有玩家的服改区被后端硬拒这条，绕不过**。两层职责分明：UX 摩擦降低误触概率，后端硬闸兜住数据面安全。

5. **面向用户文案「zone」改称「区」。** 导航、页标题、按钮、审计动作展示等面向用户的「zone 分配」改为「区分配」、「zone」改「区」（i18n key 值改写，[ADR-0033](0033-web-i18n-framework.md) 国际化框架内）。**代码标识符、`zone_assignment` 表名 / 列名、审计 action 枚举值（`zone.assign` 等）、REST 字段名一律不动**——只改面向用户的中文展示，不改契约与数据。

## 理由

- **按玩家数而非"是否在线"设闸**：空闲在线服（`PlayerCount == 0`）改区不扰动任何人，一刀切"在线即拒"会挡正常运维；以"在场有玩家"为硬拒条件，精准卡住唯一真正危险的场景。
- **排空门复用既有事实**：在线态在内存注册表、玩家数是 agent 既有上报字段，闸门只读不写、零新依赖、零 agent 改动，契合"简单优先"与控制面/数据面边界。
- **同值 no-op 先于排空门**：让"什么都没改"永远放行，既消审计噪声，也避免把幂等重试误判为危险改派。
- **后端硬闸 + 前端摩擦双层**：呼应 FR-72 的"后端兜底"哲学——UX 友好性与数据面安全各由一层负责，前端再怎么改都不破坏后端不变量。
- **只改展示不改契约**：「区」更名纯 i18n，零迁移、零兼容风险，审计与 REST 稳定。

## 后果

- `ZoneService.Assign` / `Unassign` 各多一道前置判定（查现有指派比对 + 注册表点查在线态/玩家数）；纯内存 + 一次既有仓库查询，无 N+1、无新表。
- 多一个错误码 `ZONE_SERVER_ONLINE_NONEMPTY`(409)，与既有 `INSTANCE_OFFLINE_REJECTED`(403) / `DUPLICATE_SERVER_ID`(409) / `ZONE_NOT_ASSIGNABLE_TO_BC` 并列、语义不重叠。
- 运维改一台在场服的区，须先 drain / 等排空——这是**有意的摩擦**（防误炸），不是缺陷；前端在被 409 拒时明确提示"先排空"。
- 前端归派台从"拖拽即改"变"解锁 + 显式改派 + 手输确认"，操作步数增加；这是 FR-71 的核心目标（高摩擦防误），非退步。
- 「区」更名后，旧截图 / 文档里的「zone 分配」字样与新 UI 不一致，按 doc-sync 一并更新面向用户文档。

## 备选方案

- **只加前端确认弹窗、后端不设闸**：前端始终可绕过（直接打 REST），在场服仍可被误改炸玩家，安全形同虚设。被否（必须后端纵深）。
- **"在线即拒"（不看玩家数）**：空闲在线服改区是安全的，一刀切会挡正常运维、逼运维无谓 drain。被否（以 `PlayerCount > 0` 精准设闸）。
- **改派前自动 drain 目标服再改、改完恢复**：自动排空涉及流量调度编排（[ADR-0017](0017-traffic-scheduling-decision-vs-execution.md) 明确 drain 是显式决策），且"替运维决定排空谁"超出 FR-71 范围、风险更高。被否（只读 `PlayerCount` 设闸，drain 仍由运维显式触发）。
- **保留拖拽、仅加落点确认弹窗**：拖拽本身就是低摩擦、易误触的交互；显式「改派」按钮 + 手输 serverId 复述的摩擦显著更高、更贴"删库级"防误初衷。被否。
- **把同值改派也照常落库 + 审计**：制造无意义审计噪声与无谓的覆盖链重算唤醒，且让排空门把无害的幂等操作也一并拦掉。被否（同值 no-op）。
- **改 `zone_assignment` 表名 / 审计 action 枚举值同步改成「区」**：破坏数据兼容与审计历史可追溯，无收益。被否（只改面向用户的 i18n 展示）。
