# ADR-0051：配置操作级撤回子系统（FR-116）

**状态**：已接受

## 背景

工作台原型（FR-114，纯前端 + mock）落地了「撤回 / 回滚 / 操作日志」一整套交互：操作日志逐条可见、下发 / 发布可一键撤回（回滚到操作前）、批量撤回一组下发、文件右键回滚历史版本。这套 UX 在原型里是 mock 行为——前端在内存里假装"撤回成功"，后端没有任何真实状态变化。

FR-116 要把它落成**真实后端能力**：清单内的大操作（下发 push / 反向抓取 fetch / 发布 publish）支持**事务级撤回**，撤回后控制面的配置 / 纳管真源真正回到操作前、并按需重推到在线服。这是本批最重最险的一条：它跨配置版本体系（FR-1/67）、长轮询热推（FR-2）、拓印下发门（FR-46）、反抓受管任务状态机（[ADR-0037](0037-reverse-fetch-managed-task.md)）、审计（FR-17）与命令生命周期（FR-104）多个既有子系统，且必须满足"多表写事务原子、提交成功后才触发长轮询唤醒、并发安全、幂等、不无限回溯"等硬约束。

撤回是**控制面配置 / 纳管层的操作**——它只改"事实"（配置版本指针、覆盖项、纳管记录），不写任何游戏逻辑，不引 Redis / MQ（守[架构不变量](../../.claude/rules/architecture-invariants.md) #1/#2）。

本 ADR 定义可撤回操作集、逐操作撤回语义、可逆操作记录的存储模型、以及幂等 / 并发 / 事务边界 / 过期窗口的取舍。

## 决策

### 1. 做"真撤回子系统"，不只映射已有能力

高层已拍定走**真实撤回子系统**而非把已有能力贴标签（被否的轻量方案见备选①）。新增一层**可逆操作记录**作为撤回的真源：每个大操作落地时，连同它的**反向指令 / 可逆快照**一并记账；撤回 = 读这条记录、按其反向指令在一个 DB 事务内把真源改回操作前，并按需重推。撤回本身也是一次受审计的写操作。

### 2. 可撤回操作集（清单）

纳入"会改受管真源或服务器实况"的三类大操作（与 FR-114/FR-116 清单一致）：

| 操作 | 含义 | 改了什么真源 |
|---|---|---|
| **push（下发）** | 把某层覆盖下发 / apply 到目标实例 | 触发新有效配置版本 + 长轮询热推（FR-2/FR-46） |
| **publish（发布）** | 编辑 / 上传配置后发布为某层新版本 | 配置版本前移（FR-1/67 revisions） |
| **fetch（反向抓取 ingest）** | 把扫描选定的磁盘文件入库纳管 | 新增 / 更新受管配置项 + 纳管记录（ADR-0037） |

**本期不纳入**：删除 / 重命名 / 移动等纯结构编辑操作。理由：删除 / 重命名落在配置版本体系内（删除即软删 + 新版本，[ADR-0008](0008-config-soft-delete-and-effective-md5.md)），其"撤回"天然等同于"回滚到上一配置版本"——已被 publish 的撤回语义覆盖，无需单独的可逆记录；为它们单建反向指令是镀金（守[范围纪律](../../.claude/rules/scope-discipline.md)）。若真机暴露独立诉求再按需追加，不预留空壳。

### 3. 逐操作撤回语义（复用既有体系，不另造回滚）

"回滚"一律复用既有配置版本体系（FR-1/67 revisions）与既有下发链路（FR-2 热推 / FR-46 拓印门），撤回子系统只负责**编排**，不重新发明配置存储或下发：

- **undo publish** = 把该层配置**回滚到发布前的版本指针**（复用既有版本回滚），重算有效配置 md5；若该层当前被在线服消费，则**经长轮询唤醒按需重推**（复用 FR-2，仅在有效 md5 真变时下发，承 [ADR-0008](0008-config-soft-delete-and-effective-md5.md) 决策2）。
- **undo push** = 等价于"撤销这次 apply"：把目标实例的有效配置**回滚到这次下发前的版本组合**并重推。push 不产生新配置内容、只改"哪个版本生效"，故其反向指令记的是**操作前的版本指针快照**，撤回即把指针改回 + 唤醒重推。重推**仍走 FR-46 拓印 / 下发门**（撤回不绕过既有下发安全门）。
- **undo fetch** = **撤销该次 ingest 纳管**，回到纳管前状态：复用 [ADR-0037](0037-reverse-fetch-managed-task.md) 的受管任务 / 配置版本——被该次 ingest 新建的受管项软删（[ADR-0008](0008-config-soft-delete-and-effective-md5.md) 哨兵软删，允许同标识重建）、被该次 ingest 覆盖更新的受管项回滚到 ingest 前版本。fetch 撤回的是"纳管"这件事，不删磁盘文件（agent 纯读不写，守 ADR-0037 决策8）。

撤回后若引起有效配置变化，**统一经既有长轮询"唤醒即重算比对 md5"**下发，不另建推送通道。

### 4. 新增可逆操作记录表 `reversible_operation`（GORM 可移植）

新建一张 GORM 实体记每次大操作的可逆账目。**严格可移植**（守[架构不变量](../../.claude/rules/architecture-invariants.md) #4）：状态 / 类型枚举落 `VARCHAR` + 应用层校验，反向指令快照落 `TEXT`（JSON 文本），**绝不用 MySQL 专有 `ENUM/SET/JSON` 列**，自增走 GORM 抽象，软删用哨兵默认值（承 [ADR-0008](0008-config-soft-delete-and-effective-md5.md) 决策1）。关键字段要点：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | 自增主键 | GORM 抽象，不写方言 |
| `namespace_code` / `server_id` / `scope` / `scope_target` | `VARCHAR` | 操作落在哪个环境 / 实例 / 层（global/group/zone/server）/ 目标 |
| `op_type` | `VARCHAR` | 操作类型枚举：`push` / `publish` / `fetch`，应用层校验 |
| `status` | `VARCHAR` | 状态枚举：`reversible`（可撤回）/ `reversed`（已撤回）/ `expired`（已过期不可撤回）/ `superseded`（被后续操作覆盖不可撤回），应用层校验 |
| `inverse_payload` | `TEXT` | **反向指令 / 可逆快照**（JSON 文本）：undo push 记操作前版本指针；undo publish 记发布前版本号；undo fetch 记被新建 / 被覆盖的受管项标识 + ingest 前版本，供撤回时精确还原 |
| `forward_ref` | `VARCHAR` | 关联正向产物引用（如 publish 产生的版本号、fetch 对应的 `reverse_fetch_task` id、push 对应的 `agent_command` id），便于追溯与"是否被覆盖"判定 |
| `operator` | `VARCHAR` | 操作人（与审计同源） |
| `reversed_by` / `reversed_at` | `VARCHAR` / 时间 | 撤回人与撤回时间（撤回后回填） |
| `created_at` / `updated_at` / `deleted_at` | 时间 | 时间戳；`deleted_at` 哨兵软删 |

记录在大操作的**同一 DB 事务内**与正向写一并落库（操作与其可逆账目原子，不存在"操作成功但没记账→无法撤回"的窗口）。

### 5. 事务边界：撤回的多表写原子，提交成功后才唤醒

撤回涉及的多表写（改 `reversible_operation.status` + 回滚配置版本指针 / 软删受管项 + 写审计）**必须在一个 DB 事务内原子完成**（守[质量底线](../../.claude/rules/testing-and-quality.md)：发布 / 回滚 / 改派等多表写在事务内原子）。**长轮询唤醒只在事务提交成功后触发**——撤回引起的有效配置变化，按 scope 算最小受影响集合再唤醒，被唤醒的 agent 重跑解析比对 md5、真变才下发（复用既有长轮询机制，承 [ADR-0008](0008-config-soft-delete-and-effective-md5.md)）。事务内**不做任何下发 IO**；DB IO 一律在 runtime 三锁（Registry/Hub/Health）之外（守锁外 IO 不变量）。

### 6. 幂等：同一操作重复撤回只生效一次

撤回端点幂等：以 `reversible_operation.status` 作幂等闸——只有 `reversible` 状态可被撤回；撤回成功后状态原子置 `reversed`。重复点击 / 重试同一撤回，第二次看到状态已非 `reversible`，**直接返回"已撤回"成功响应、不再执行任何回滚写**（幂等成功，非报错）。状态翻转与回滚写在同一事务内，杜绝"两个并发撤回都读到 `reversible`、都执行回滚"的双撤回。

### 7. 并发：撤回与新操作 / 其它撤回并发安全

并发安全靠 `store.RunInTx` + **行级锁 / 乐观锁**根因解决（沿[并发缺陷根因修复](../../.claude/rules/testing-and-quality.md)纪律，不在测试里容忍）：

- 撤回事务内先以行级锁（`SELECT ... FOR UPDATE` 经 GORM 抽象，或 `version` 列乐观锁 CAS）锁定目标 `reversible_operation` 行，再校验状态 + 执行回滚 + 翻转状态，全程串行化同一行的并发撤回。
- **撤回 vs 新操作竞争**：撤回与"针对同一 scope 的新 push/publish/fetch"在配置版本指针上竞争——以配置版本表自身的版本 CAS / 行锁保证后写覆盖前写，二者各自原子；撤回若发现目标版本指针已被后续操作前移（见决策 8 `superseded`），则拒绝撤回而非脏写。
- 集成测试以真实 MySQL `-count` 反复跑撤回 / 新操作 / 多撤回交错，验非脆。

### 8. 过期窗口：超时或被覆盖即不可撤回

为避免无限回溯与脏撤回，可逆记录有**有限可撤回窗口**：

- **时间窗过期**：`reversible` 记录创建超过 N（可配置，经运维设置 store 热改，承 [ADR-0038](0038-ops-settings-store-hot-reload.md)；默认值入应用层常量、不硬编码散落）后，由既有清理器（复用 / 扩展 ADR-0037 命令清理器思路）扫描并置 `expired`、清空 `inverse_payload` 瞬态（避免快照 TEXT 长期滞留）。
- **被覆盖失效**：同一 scope 上若已发生后续操作（依 `forward_ref` 与配置版本指针判定该记录的版本已非当前），该记录置 `superseded`、不可撤回——撤回一个已被覆盖的旧操作会把后续操作一并抹掉，属脏撤回，禁止。撤回端点对 `expired` / `superseded` 返回明确错误（提示该操作已超时 / 已被后续操作覆盖，不可撤回）。

### 9. 操作日志：复用既有审计 + 命令生命周期呈现，撤回动作入审计

操作日志**不新建独立日志表**：正向大操作沿用既有审计（FR-17，`config.publish` 等 action）+ 命令生命周期（FR-104，`agent_command` 下发 / 拉取 / 回执）呈现；可逆记录（决策 4）提供"哪些可撤回 + 状态"的撤回维度。**撤回动作本身写入既有审计**（新增 `config.undo-push` / `config.undo-publish` / `config.undo-fetch` 等 action，敏感内容不入 detail），管理台复用审计页 / 命令观测页过滤可查，不另起 UI 子系统。

### 10. 守边界：控制面只存事实、不引重型件

撤回是控制面配置 / 纳管层操作，只改"事实"（版本指针 / 覆盖项 / 纳管记录），**不写任何游戏逻辑**（守[架构不变量](../../.claude/rules/architecture-invariants.md) #1）；**不引 Redis / MQ / 事件溯源框架**（守 #2，详见备选③）。分层 `router → handler → service → repository` 单向：撤回编排在新 `ReversibleOperationService`（复用既有版本回滚 / 反抓任务 / 下发服务），handler 不碰 GORM / 内存结构。

## 理由

- **真撤回比贴标签有表达力**：被否的轻量方案（备选①）只能"回滚到上一配置版本"，无法表达"撤销一次反抓 ingest 纳管"或"批量撤回一组下发"——这两个正是 FR-114 原型 UX 的核心，用户选了真撤回。
- **复用而非另造**：回滚复用 FR-1/67 版本体系、纳管撤销复用 ADR-0037、重推复用 FR-2/FR-46、日志复用 FR-17/FR-104。撤回子系统只新增"可逆账目 + 编排 + 撤回端点"这一薄层，其余全是已验证的成熟机制——最小代价、不重复造轮子（守简单优先）。
- **可逆账目独立成表**：把反向指令记成独立实体（而非塞进审计 detail 或命令表）让"哪些可撤回 / 撤回语义 / 过期"有清晰真源、可单测、可清理，不污染审计与命令表（与 ADR-0037 "任务独立实体、命令仍作原语"同一思路）。
- **过期 + 被覆盖双闸**：时间窗防无限回溯、被覆盖防脏撤回——只有"近期且未被覆盖"的操作可撤回，符合直觉且杜绝把后续操作一并抹掉的危险。
- **事务 + 幂等 + 行锁三件套**是 Beacon 既定的多表写纪律（发布 / 回滚 / 改派同款），撤回直接沿用，不发明新并发模型。

## 影响

- **新增**：一张 `reversible_operation` 表 + `ReversibleOperationService`（编排撤回 + 幂等 + 过期判定）+ 撤回端点（`POST /admin/v1/.../undo`，full 角色写、readonly 403、入审计）+ 大操作落地处补"记可逆账目"（与正向写同事务）+ 清理器扫过期。新增 `config.undo-*` 审计 action。
- **复用不改**：FR-1/67 版本回滚、FR-2 长轮询热推、FR-46 拓印下发门、ADR-0037 反抓受管任务 / 软删、FR-17 审计、FR-104 命令观测——撤回是其上的编排层，不改它们的契约。
- **运维设置**：可撤回窗口 N（小时）作为热改项进 store（ADR-0038），默认值入常量。
- **agent 零改**：撤回全在控制面完成（改版本指针 / 软删受管项 + 唤醒重推），agent 仍只消费有效配置 / 文件树，无新命令、无 jar 重建。
- **测试**：merge 仍纯函数；高风险区补"撤回幂等 / 撤回与新操作并发 / 多撤回交错 / 过期与被覆盖拒撤回 / 撤回多表写原子"的单元 + 真 MySQL 集成（`-count` 验非脆）+ 真机过。
- **关联**：FR-116（本 ADR 对应需求），依赖 FR-111（双面板工作台）；落地 FR-114 原型 UX 的真后端。

## 备选方案

- **① 仅映射已有能力，不做操作级撤回子系统（被否的轻量方案）**：把"回滚"直接等同既有配置版本回滚、"操作日志"直接等同审计页，不新增可逆账目与撤回端点。优点是零新表、零新服务。**否**：表达力不足——无法"撤销一次反向抓取 ingest 纳管"（纳管不是单纯版本回滚）、无法"批量撤回一组下发"（审计页只读、不能逆向执行）。用户明确选了真撤回子系统而非贴标签。记录在案。
- **② 前端乐观撤回、后端只留审计**：前端在内存里"假装撤回"、后端只记一条审计、不真改真源。优点是后端几乎零改。**否**：非真撤回——前端显示"已撤回"但控制面真源 / 在线服实况未变，状态不一致、刷新即穿帮，正是 FR-114 原型要被替换掉的 mock 行为。
- **③ 事件溯源 / 通用 undo 框架**：把所有操作建模为事件流、撤回 = 重放到某个事件点 / 生成补偿事件，引入通用 undo 引擎。优点是理论上任意操作可撤回。**否**：重型件，违背简单优先与[架构不变量](../../.claude/rules/architecture-invariants.md) #2（MVP 禁分布式一致性 / 重型框架）——Beacon 只需对三类明确大操作做有限窗口撤回，事件溯源是用大炮打蚊子，且与现有"配置版本指针"真源模型割裂、迁移成本高。用有限的可逆账目 + 复用既有版本体系即足够。
