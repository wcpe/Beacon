# ADR-0001：自研控制面，而非直接使用 Nacos

**状态**：已接受

## 背景

需要一个面向 MC 集群（50 服起步、持续扩大区）的配置中心 + 服务发现。Nacos 本身具备配置中心、服务发现、配置灰度、历史/回滚等能力，是现成选项。

## 决策

**自研一个轻量控制面**，只实现 MC 真正需要的子集，用 MC 原生概念（namespace/大区/zone/serverId）建模；不引入 Nacos 整套。

## 理由

- Nacos 偏重（需自带集群 + MySQL），其 `namespace/group/dataId` 模型要硬映射到 MC 的环境/大区/zone，且运维需额外学习 Nacos。
- MC 特有需求（zone 指派权威、合区作为配置对象、按 serverId 解析覆盖链）在自研里表达更自然。
- "版本发布编排"等（排空→换 jar）本就不在 Nacos 能力内，无论如何要自己写。
- 50 服规模，自研轻量控制面成本可控；full Nacos 反而是过度工程或仅是捷径。

## 后果

- 配置历史/回滚/灰度/推送需自己实现，是真实工作量（但 MVP 只做配置中心 + 发现 + 健康，范围受控）。
- 借鉴 Nacos 验证过的**模式**（客户端本地快照 fail-static、长轮询推送、配置版本表），不照搬实现。

## 备选方案

- **直接跑 Nacos + 写 MC 适配层**：省掉配置/历史/回滚，但引入重型依赖与模型错配，且编排能力仍需自研。被否。
