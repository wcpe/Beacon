# ADR-0024：bc 上报自身后端归属作为控制面只读事实（仅内存、随注册/心跳更新）

**状态**：已接受

## 背景

集群拓扑可视化（[FR-37](../PRD.md)）要画出真实的 **bc（bungee 代理）→ bukkit（后端子服）** 连线，需要一份「哪个代理当前代理了哪些后端」的事实。控制面当前只有逐实例的注册/健康事实（serverId / role / address / zone / 负载指标），**没有任何一处记录代理与后端的从属关系**：

- 控制面经 `zone_assignment` 知道每个 bukkit 子服归哪个 zone，但**不知道哪台 bc 实际把它列在了自己的服务器目录里**——同一 zone 的子服可能被多台代理共享，连线关系只有代理自己清楚。
- agent 侧 BeaconAgentProxy 已经维护着一份「当前代理的后端子服目录」（`ProxyServerDirectory`，含 Beacon 注入 + 手工配置的子服），这正是连线事实的权威来源，但它只活在 bc 进程内、控制面看不到。

需要让 bc 把「自身后端归属」这份事实上报给控制面，且**不能破坏既有边界**：控制面只存事实不写游戏逻辑（[architecture-invariants](../../.claude/rules/architecture-invariants.md) #1）、agent 自管身份（#5）、注册/健康真源在进程内存（#3、[ADR-0003](0003-no-redis-in-mvp.md)）。同时严守范围纪律——只搬「事实」，不做任何跨服玩家行为（看人 / 引流属业务插件，[scope-discipline](../../.claude/rules/scope-discipline.md)）。

## 决策

bc 把「当前代理的后端子服 serverId 集合」作为**自身上报的只读事实**附加到既有 register / report 请求，控制面存进内存 `Registry`：

1. **事实由 bc 自报、agent 自管**：bc 的 `backends` 取自 agent-core `ProxyServerDirectory.backendServerIds()`（BungeeCord 服务器目录全部 keys，即本代理实际能转发到的后端），由 agent 自行采集上报——与「agent 自管身份」一脉相承，控制面不反向推断、不指派后端归属。
2. **仅 bungee 填，bukkit 不填**：`backends` 是代理特有事实；bukkit 子服无后端，恒不上报该字段。
3. **可选字段、向后兼容**：`backends` 为附加字段，旧 agent / bukkit 不发即缺键。register 缺键 → 空集；report **用「缺键 vs 显式空集」区分**——旧 agent / bukkit 缺键时控制面保留原集合不动，bc 显式上报（含空集即清空）才刷新。
4. **存为内存事实、不落 DB**：`backends` 进 `runtime.Instance`（与 role / address / 负载指标同列实例事实），随注册/上报刷新，**与注册/健康真源同在进程内存**——重启即随实例重新注册重建，不落 MySQL（守真源切分：注册/健康真源=内存）。`Registry` 锁内更新、写入前深拷贝与调用方隔离，**不涉任何 DB IO**。
5. **只读暴露供拓扑消费**：实例视图（`GET /admin/v1/instances`）输出 `backends`，供 FR-37 拓扑端点据此画 bc→bukkit 连线。控制面只展示这份事实，不据它做任何调度 / 连接决策。

## 理由

- **事实而非逻辑**：`backends` 是「这台代理此刻代理了哪些后端」的客观陈述，和 role / address 一样属注册事实，落控制面不越「只存事实」的边界；连线的可视化是纯展示，不含任何游戏逻辑。
- **复用既有上报通道、最小新增面**：搭既有 register / report 信封顺带上报，不新增端点、不新增表、不引中间件（守简单优先 / 禁重型件）；agent 侧复用已存在的 `ProxyServerDirectory`，采集零额外开销、在既有异步上报线程内完成、不上 MC 主线程。
- **内存真源一致**：后端归属是注册态的派生事实，生命周期与实例注册绑定，落内存与注册/健康同源、重启自然重建，避免再引一张表去同步一份本可随注册重报的瞬时事实。
- **向后兼容**：字段可选，旧 agent 不发、旧控制面忽略；report 的「缺键不动 / 显式才刷新」语义保证混部期 bukkit 与旧 bc 的上报不会误清空 bc 已报的后端集合。

## 后果

- `runtime.Instance` **增 `backends []string` 字段**（仅内存）：bc 实例多一份后端集合事实，bukkit 恒空；随注册/上报刷新、重启清零并随重新注册重建。
- agent register / report 载荷**各增可选 `backends` 键**（仅 bc 非空填充）：信封只增不改，旧 agent 不发、旧控制面忽略，向后兼容。
- 后端集合是**上报那一刻的快照**：代理目录在两次上报之间的变化要到下一次心跳/上报才反映到控制面，存在与采样间隔同量级的滞后——对拓扑可视化（趋势性展示）可接受，**不作精确实时连接依据**。
- **边界保持**：本 ADR 只搬「后端归属事实」，不引入任何跨服玩家行为；若未来要据此做引流 / 看人 = 越界，须先立新 ADR。

## 备选方案

- **控制面据 `zone_assignment` 反推连线**：用「同 zone 的 bc 与 bukkit 即相连」近似。但同 zone 子服可能被多台代理共享、代理也可能配置手工后端，反推得到的是「应该可连」而非「实际代理了」，与真实拓扑不符。落选（要真实连线就得由代理自报）。
- **后端归属落 MySQL 表**：可跨重启留存历史。但它是注册态的派生瞬时事实、随实例重连即重建，落表要额外同步逻辑且与「注册/健康真源在内存」相悖，撞简单优先。落选（内存随报随存足够）。
- **新增专用上报端点 `POST .../backends`**：语义更聚焦，但平白多一条端点与一轮网络往返，而该事实天然可搭既有 register / report 顺带上报。落选（复用既有信封）。
