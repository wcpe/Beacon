# ADR-0029：文件树结构化文件跨层深合并（取代 ADR-0010 决策1）

**状态**：已接受

> 本 ADR **取代 [ADR-0010](0010-file-tree-hosting-blob-channel.md) 决策1**（「整文件 blob：scope 覆盖 = 整文件覆盖，不深合并」）。ADR-0010 其余决策（2 增量同步 / 3 MySQL TEXT 存储 / 4 镜像落盘 / 5 fail-static）继续有效。

## 背景

ADR-0010 决策1 定下：文件树托管（通道B）按 `path` **整文件覆盖**，覆盖链上取层级最高的那一整份，**绝不深合并**。理由是当时面向 `.allin`/`.js`/`lang` 等整文件语义的资源，深合并无意义。

但 FR-44 暴露真实场景：100+ 异构服（小游戏/战斗/大厅，靠大区/小区逻辑分组）要共享一份**结构化配置基线**（`.yml`/`.json`/`.properties`），各层只想叠加少量增量。整文件覆盖下，小区改 3 个键也得维护整份副本——而通道A（配置中心）早有键级深合并，却因第三方插件**无法接入 agent API** 而用不上。结论：结构化配置文件**也需要**通道A 那样的深合并，只是要长在通道B 的"镜像落盘、第三方零改"形态上。

## 决策

通道B 文件树解析（`internal/filetree.Resolve`）按 `path` 后缀与一个豁免标记**分流**：

1. **结构化文件深合并（仅多层）**：后缀 `.yml`/`.yaml`/`.json`/`.properties` 的文件，**当某 path 有 ≥2 层贡献时**跨 global←大区←小区←单服 四层**按键深合并**，**复用 `internal/merge`**（标量覆盖 / map 深合并 / list 整体替换 / 高层显式 `null` 删键 / 固定键序与 md5）——与通道A 同一套合并语义与幂等保证。**单层贡献则字节原样透传**（不 parse/reserialize），避免对无需合并的文件做有损往返（值归一化见后果）。
2. **非结构化整文件兜底**：其余后缀维持 ADR-0010 原行为（取覆盖链最高层那一整份），**绝不深合并**。
3. **按文件整文件豁免（path 级）**：`file_object` 增 `WholeFileOverride` 标记；**任一层**标真则整 path 走整文件覆盖（取最高层那一整份、保注释、不被重渲染打乱），不由 winner 单层独断，与遍历顺序无关。
4. **控制面渲染、agent 零改**：控制面算出每服合并后的整文件，经既有 `files/manifest`（md5=合并后整文件 md5）/`files/content`（合并后整文件）下发；agent 仍是哑镜像、原子落盘、fail-static（ADR-0010 决策 2/4/5 不变）。
5. **坏内容降级**：某层结构化内容解析失败时，**该 path 回退为整文件取 winner**（`Resolve` 保持无副作用纯函数、确定性，一坏文件不拖垮整棵树），不抛错中断整树解析。
6. **发布期解析校验**：结构化文件（yml/json）在发布 / 导入前做 `merge.Parse` 解析校验，坏语法即拒入库（`CONTENT_SCHEMA_INVALID`，与通道A 发布校验对齐），使决策5 的运行期回退成为兜底而非常态。只校验「能解析」、不强制顶层 map（文件树可托管顶层 list/scalar，与通道A 的 schema 校验有别）。

## 理由

- **同一套合并引擎**：直接复用 `internal/merge`，不另写第二套合并逻辑（杜绝双实现漂移），固定键序保证 md5 幂等、长轮询比对可靠（沿用 [ADR-0008](0008-config-soft-delete-and-effective-md5.md) 思路）。
- **第三方零改**：合并在控制面完成、agent 落地整文件，插件照旧从磁盘读——把通道A 的能力带给"接不进 agent API"的第三方插件，正是 FR-44 的核心价值。
- **整文件语义不丢**：非结构化兜底 + 豁免标记保住"整文件覆盖"这一原有正确用法（脚本、注释敏感文件）。
- **纯函数、可穷举测**：合并仍是无副作用纯函数，便于穷举单测覆盖标量/map/list/null/混合层（守 `testing-and-quality` 必测高风险区）。

## 后果

- `internal/filetree` 依赖 `internal/merge`（同为无副作用纯逻辑包、无环）。
- `model.FileObject` 增 `whole_file_override`（基础布尔、零方言、可切 Postgres；AutoMigrate 加列，既有行默认 false）。
- **向后不兼容 + 值归一化（已与用户确认）**：**单层**结构化文件字节原样透传、不受影响（决策1 短路）；**多层**结构化文件按键合并并 **parse→reserialize 重渲染**——这不仅丢注释 / 原键序，还会**归一化值**：`007`→`7`（前导零）、`2026-06-22`→`2026-06-22T00:00:00Z`（日期转时间戳）、`1.10`→`1.1`（版本号当浮点）、JSON 大整数精度丢失（`…678`→`…680`）。故**多层**结构化文件升级后生效结果可能变；不能容忍值归一的文件须在某层标 `wholeFileOverride`（= 整文件覆盖、放弃该 path 的合并）。现网升级前复核多层结构化托管文件。（此条经发版前 code-review 细化：单层透传 + 发布校验已大幅收窄受影响面。）
- ADR-0010 决策1 标记"已被 ADR-0029 取代"；其余决策不变。
- 后续 FR-45（有效文件树预览/逐键来源）、FR-46（拓印审核台）建在本合并模型之上。

## 备选方案

- **维持整文件覆盖（ADR-0010 决策1）**：第三方插件无法"基线+增量"，每层维护整份副本。被 FR-44 否决。
- **引保注释 yaml 库做合并**：可不丢注释，但增重型依赖、且跨层合并后注释归属本就语义不清；本期不引，用"整文件豁免"满足注释敏感场景。否决。
- **在 agent 侧合并**：违 agent 哑/零改与控制面渲染边界，且要把 merge 引擎搬进 Kotlin（双实现漂移）。否决——合并留控制面。
- **坏内容时整树解析报错**：一个坏文件会让该服整棵文件树拉取失败。否决——改为该 path 回退整文件 winner 的局部降级（决策5）。
