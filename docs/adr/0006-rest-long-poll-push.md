# ADR-0006：配置推送用 REST 长轮询（"唤醒即重算比对"）

**状态**：已被 [ADR-0015](0015-sse-server-push-transport.md) 取代（决策已转向单条 SSE 推送流；长轮询为迁移前的运行实现，实际迁移随 FR-24 实施，在此之前长轮询仍有效。正文保留不改）

## 背景

配置变更需要近实时推送到 agent 实现动态热更。可选：长轮询、WebSocket/SSE、消息队列、Redis pub/sub。已决 MVP 不引 Redis/MQ（[ADR-0003](0003-no-redis-in-mvp.md)）、协议用 REST。

## 决策

用 **REST 长轮询**，机制为 **"唤醒即重算比对"**：

1. agent 带当前有效配置 md5 发起 `GET .../config/effective`。
2. 服务端**先注册 waiter，再算当前 md5**：不等立即返回；相等才挂起（含超时 + 客户端断连）。
3. 发布/回滚/改派**事务提交后**按 scope 算最小受影响 serverId 集合，仅唤醒集合内 waiter。
4. 被唤醒后**重跑解析比对 md5**，真变才下发，未变继续挂起。
5. 超时返回 304，agent 续杯。

## 理由

- 长轮询是 Nacos 验证过的成熟方案，纯进程内内存通知即可，无需中间件。
- "先注册后算" 顺序消除"注册前发布丢唤醒"窗口（生产真实触发的竞态）。
- "唤醒即重算比对" 保证只在有效配置**真正变化**时才推送——覆盖链下（改的键可能被高层覆盖）避免无意义下发的关键。
- 受影响集合按 scope 精确缩小（global→全 ns、group→该 group、zone→反查分配、server→单点），不广播全员。

## 后果

- 控制面重启丢挂起请求，agent 退避重连恢复（可接受）。
- Registry/Hub/Health 三锁独立、DB IO 在锁外，结构上无死锁。

## 备选方案

- **WebSocket/SSE 长连接推送**：可行，但相比长轮询多一套连接状态管理，MVP 收益不大。备选保留，未采用。
- **Redis pub/sub**：违背 [ADR-0003](0003-no-redis-in-mvp.md)。被否。
