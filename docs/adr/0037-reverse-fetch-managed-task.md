# ADR-0037：反向抓取受管任务 + 状态机 + 两段式 scan/submit

**状态**：已接受

## 背景

[ADR-0027](0027-reverse-fetch-channel-and-security.md) 定的反向抓取是**一次性命令**：admin 触发 → agent 读整棵 `plugins/` 树、过滤、**一次性回传所有文本文件内容** → 控制面 ingest 落库。它的**决策5**为防"半截覆盖污染基线"采取**超限整批失败**：单文件 / 文件数 / 总字节任一超限 → agent 不上传、控制面拒入、命令 `failed`，**绝不部分落库**。

真机暴露这套**整批失败**会被运行时垃圾文件击穿：`plugins/` 下混着插件运行时数据（`*.jsonl` 指标、ServerProbe 落盘），**单个就可能超 1MB 单文件上限** → agent 侧 `PluginsTreeFilter` 判 `Rejected` → **整批不传** → 反向抓取 100% 失败，连同目录里真正想要的小配置文件一起捞不上来。加大上限不是治本（运行时文件可以任意大、且本就不该入配置库）。

根因是**抓取粒度太粗**：一次命令 = 整树读内容 + 整批落库，没有"先看清单、人工挑、只抓选定"的中间环节，于是无关大文件绑架了整次抓取。真正的治根是把反向抓取从"一次性命令"升级为**受管任务 + 两段式（先扫清单、再抓选定）**，让超限的运行时垃圾在**扫描清单**里被看见、被排除，而非把整批拖垮。

本 ADR **取代 ADR-0027 决策5**（超限整批失败 → 扫描列出不失败 + 提交选定集 + 超阈值须确认）、**扩展决策1**（一次性命令 → 受管任务 + 两段式命令）；ADR-0027 其余决策（2/3/4/6/7/8 的安全边界）**全部保留不变**。自身目录托管由 [ADR-0028](0028-allow-hosting-agent-self-dir.md) 已解决（与本 ADR 无关）。

## 决策

1. **反向抓取升级为「受管任务」，新建 `reverse_fetch_task` 实体 + 状态机。** 新表 `reverse_fetch_task`（GORM 可移植：`VARCHAR` 状态、`TEXT` 存清单、软删哨兵、不用 `ENUM/JSON` 列），持有：`namespace_code` / `server_id` / `scope`(group/server) / `group_code` / `scope_target` / **状态** / **扫描清单(manifest，TEXT)** / 选定 path 集 / 计数(总/选定/忽略/超阈值/跳过) / 命令引用 / `operator` / 时间戳。状态机：
   ```
   scanning → pending-review → fetching → ingesting → done
                  │                │           │
                  └────────────────┴───────────┴──→ failed / cancelled / expired
   ```
   `scanning`(已下发 scan 命令待 agent 回清单) → `pending-review`(清单到、待人工审核选定) → `fetching`(已下发 submit 命令待 agent 回选定内容) → `ingesting`(选定内容到、落库中) → `done`；任一阶段失败 `failed`、人工取消 `cancelled`、超时 `expired`。任务**进度**由状态 + 计数表达、可查询。

2. **`agent_command` 仍作底层命令原语，任务编排两次下发。** 不把整个任务塞进一条命令——任务在 `scanning` 下发一条 **scan 命令**(`mode=scan`)，在 `fetching` 下发一条 **submit 命令**(`mode=submit` + `selectedPaths`)。命令仍复用既有 SSE 唤醒 + `agent_command` 落库 + CAS 状态迁移（ADR-0027 决策1 的通道不变），只是从"一条命令走完"变成"任务驱动两条命令"。任务与命令一对多、任务是真源、命令是它的执行手段。

3. **两段式：scan 只列清单不读内容；submit 只回传选定内容。**
   - **scan 阶段**：agent 遍历 `plugins/` 树**只采元信息**——每文件 `path` + `size` + `isText`(UTF-8 文本判定) + `overThreshold`(是否超单文件阈值)，**绝不读取文件内容、绝不因任何文件超限而失败**。回传清单 → 控制面存入任务 manifest → 任务转 `pending-review`。超限的运行时垃圾在此被**列出并红标**，而非拖垮扫描。
   - **submit 阶段**：人工审核选定一个 path 子集后，控制面下发 `mode=submit` + `selectedPaths` → agent **仅读取并回传这些选定 path 的内容**（不再整树回传）→ 控制面 ingest 落库 → 任务转 `done`。

4. **超限不再整批失败（取代决策5）。** scan **永不失败**（无内容、清单可大，不受单文件/文件数/总字节上限约束）。submit 阶段：**超单文件阈值的文件红标，须显式确认才能纳入选定集**（前端 FR-60 红标 + 确认，后端校验"选了超阈值文件必须带确认标记"否则拒该文件——**只拒该文件、不拒整批**）；文件数 / 总字节上限仍作 **submit 兜底**（防一次提交异常巨大），但不再连累 scan、也不再把整批一票否决。**绝不静默跳过**：超限文件始终在清单里可见，纳入或排除都由人显式决定（契合"完整治根、不止血"）。

5. **单实例同时仅一个活跃任务（互斥）。** 每 `(namespace, server_id)` 同时**至多一个非终态**（`scanning`/`pending-review`/`fetching`/`ingesting`）任务。新建任务前查有无活跃任务，命中 → 拒 `409 REVERSE_FETCH_TASK_ACTIVE`（提示已有活跃任务及其 id/状态，引导先完成或取消）。以"活跃任务唯一"的 DB 查询 + 唯一约束保证，杜绝同一台服并发多个抓取互相踩。

6. **agent 两段式实现 + 架构约束不破。** agent-core 命令载荷扩 `mode`(scan/submit) + `selectedPaths`；新增 scan 路径（只 `stat` 取大小 + UTF-8 文本判定，**不读内容**）与 submit 路径（按 `selectedPaths` 子集读内容回传）。全程仍在 **async 线程**、**绝不碰 MC 主线程**（读盘是阻塞 IO，沿 ADR-0027 决策8）；HTTP/JSON 仍只在适配器层（[ADR-0005](0005-agent-transport-codec-abstraction.md)）；agent 仍**纯读不写盘**、**最终权威 + 控制面再校验**双保险不变；agent 自身目录 observe-only 写回守卫（ADR-0028）不变。**改 agent-core → 双端 jar（bukkit/bungee）重建并真机重部。**

7. **生命周期：可查 / 可取消 / 会过期。** 任务可经 admin 端点查询（含状态 + 进度 + 清单）；任一非终态可人工取消 → `cancelled`；scan 回清单超时 / 审核停留超时 / submit 回内容超时 → `expired`；过期清理（复用并扩展既有命令清理器）扫陈旧非终态任务转 `expired` 并清空 manifest 瞬态（避免清单 TEXT 长期滞留）。

8. **ADR-0027 安全边界全部保留。** 仅 ingest `plugins/` 根下文本配置、禁符号链接逃逸 / `..` / 绝对 / UNC（决策3）、排除 `.jar` 与二进制（决策4）、agent 最终权威 + 控制面再校验（决策6）、命令鉴权(full 角色) + 审计、detail 不含文件内容（决策7）、agent 仅 async 读盘不主动不写（决策8）——一条不松。

## 理由

- **两段式把"看"和"抓"分开**：先扫元信息清单（轻、永不失败、看得见全貌），人工排除运行时垃圾，再只抓选定——粒度从"整树整批"降到"逐文件可选"，超限垃圾再大也只是清单里一个红标项，绑架不了整次抓取。这是对决策5"整批失败"的真正治根。
- **受管任务 + 状态机**让"扫描→审核→抓取→入库"全程可见、可查、可取消、可审计、可恢复（跨 SSE 断连），比一次性命令的黑盒"成功/失败"强得多，也是承载 FR-59 审核 / FR-60 任务台的底座。
- **任务与命令分层**：命令原语（SSE 唤醒 + 落库 + CAS）成熟稳定、不动；任务在其上编排两段，职责清晰、可单测，避免把状态机硬塞进"一命令一回合"的 CAS 语义里拧巴。
- **互斥单活跃任务**：抓取是低频重操作，同一台服并发多个抓取毫无意义且互相踩清单 / 选定，单活跃约束最省心。
- **改 agent 不可免但收敛**：scan/submit 是 agent 读盘行为的本质变化，控制面无法替 agent 决定"只列清单"或"只读子集"；但改动收敛在 agent-core 的命令分路 + 读盘两条路径，安全边界与线程模型一条不破。

## 后果

- 多一张 `reverse_fetch_task` 表 + 一个 `ReverseFetchTaskService`（编排状态机 + 互斥 + 两段命令 + ingest）；`agent_command` 加 `mode=scan/submit` 与 `selected_paths` 载荷、加一个 scan 清单回传端点。
- agent-core 改动（命令载荷 + scan/submit 两条读盘路径）→ **双端 jar 重建 + 真机重部**；旧 agent 与新控制面的兼容：控制面只对**新版 agent**下发 scan/submit（旧 agent 收不到新 mode 即按旧逻辑——但新工作流要求新 agent，真机须双端升级）。
- 反向抓取从"一键整批"变"扫描→审核→提交"三步，操作步数增加（这是 FR-58~60 的核心目标：可控、可审、治根），FR-60 前端审核台 / 任务台承载这套交互。
- 超阈值文件不再被静默拒，而是清单红标 + 须确认——运维看得见、决定权在人。
- ADR-0027 决策5 作废（标记被本 ADR 取代）、决策1 扩展；ADR-0027 文件正文不改（不可变），仅在其状态区注明决策5被 ADR-0037 取代。

## 备选方案

- **只加大单文件 / 总量上限**：运行时文件可任意大、且本就不该入配置库，加大上限是把垃圾也吞进来，不治本。被否。
- **agent 静默跳过超限文件、其余照传（止血）**：改动小、能让真机不再整批失败，但**静默跳过**违背"完整治根、不止血"——运维不知道哪些被跳了、也没有"看清单挑选"的受管流程，且无法表达"我就是要这个大文件"。被否（用户明确要受管两段式而非止血）。
- **不建任务表、把状态机塞进 `agent_command`**：一条命令承载"扫描→审核→提交"整个任务，但命令是"一回合一 CAS"的原语，硬塞两次下发 + 长审核停留的状态机会把命令语义拧巴、清单 TEXT 也污染命令表。被否（任务独立实体、命令仍作原语）。
- **scan 也带内容（一次抓全、前端再筛）**：等于没解决——整树读内容回传仍会被超限文件击穿、且把不要的运行时垃圾全搬到控制面再筛，浪费带宽与存储。被否（scan 只列元信息不读内容）。
- **两段式纯控制面实现、不改 agent**：控制面无法让 agent"只列清单"或"只读选定子集"，agent 永远整树读内容回传，治根无从谈起。技术上不可行。被否。
