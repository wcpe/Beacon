# ADR-0041：告警事件持久化实体 alert_event（留痕 + UI 信息流，与 FR-28 推送通道并存）

**状态**：已接受

## 背景

FR-28（[ADR-0019](0019-health-alert-channel-abstraction.md)）把健康告警出口抽象为 `Alerter` 接口、`Dispatcher` 扇出，第一版实现**站内信**（`InboxAlerter`，进程内环形缓存、重启清零）与 **webhook**（`WebhookAlerter`，HTTP POST 往外推）。这两个通道都是「**往外推**」：webhook 推给外部系统、站内信仅留进程内最近 N 条且重启即失。

FR-89 暴露的盲区：**控制台没有任何告警历史 / 事件信息流**。运维想回看「上周哪台服掉过线、什么时候恢复的」时无处可查——站内信重启清零、webhook 推走就不在本地。需要一处**可跨重启留存、可按类型 / 级别 / 时间过滤回看**的告警事件信息流。

ADR-0019 当时明确「告警不落库」，并把「告警落 MySQL 表 + 管理台查库」列为**被否备选**（理由：健康真源在内存、告警是其派生，落库引入第二权威）。FR-89 的诉求与该决策的前提不同——这里**不是把告警当健康真源的第二权威**，而是把「发生过的告警事件」作为**只读历史留痕**持久化，供 UI 信息流。需以新 ADR 明确这一边界、并取代 ADR-0019「告警不落库」那一条结论。

## 决策

1. **新增 `alert_event` 实体**持久化告警事件，与 FR-28 既有通道**并存、不替代**：
   - 通道（webhook / 站内信）职责不变 = 「**往外推 / 进程内即时**」；
   - `alert_event` 表职责 = 「**留痕 + UI 信息流**」，跨重启可查。
2. **以新增一个 `Alerter` 通道（`PersistAlerter`）接入**，而非改 `HealthScanner` 或既有通道：`PersistAlerter.Notify` 把一条告警落 `alert_event` 表。这样**复用 ADR-0019 的扇出抽象**——告警触发点（当前为健康流转 degraded/lost/offline）零改动，任何现有 / 未来的 `Dispatch` 调用都自动留痕；新增告警类型（发布失败 / 后端不可达等）将来在各自发生处调 `Dispatch` 即自动持久化，无需再碰本通道（守 OCP）。
3. **DB IO 在锁外、失败仅 WARN 不阻断**：`Dispatch` 由健康扫描循环在**任何运行态锁之外**触发（ADR-0019 已保证）；`PersistAlerter.Notify` 落库失败只返回 error 交由 `Dispatcher` 记 WARN，绝不阻断扫描 / 其它通道（与 webhook 同精神）。
4. **可移植 GORM**：`alert_event` 全部基础类型——类型 / 级别 / serverId / namespace / message 落 `VARCHAR`（枚举应用层校验）、detail 落 `TEXT`、自增主键经 GORM 抽象、`created_at` 由全局 `NowFunc` 统一落 UTC。**禁 JSON/ENUM 列与方言专有 SQL**，可切 Postgres。
5. **本期范围（守范围纪律 / YAGNI）**：只做「**持久化既有告警事件 + 列表端点 + UI 信息流**」。**不**新增告警规则 / 不改 FR-28 通道行为 / 不做告警确认（ack）工作流 / 不做告警去重静默窗口。事件类型常量预置 `health-transition`（当前唯一真实触发）并保留 `publish-fail` / `backend-unreachable` 作为**类型枚举取值**（供将来在对应发生处接 `Dispatch` 时复用），但**本期不凭空制造这两类触发点**。

## 理由

- **与 ADR-0019 抽象同构、零改触发点**：落库做成又一个 `Alerter`，扇出 / 兜错 / 锁外 IO 全部复用既有机制，是改动最小、最不破坏现有架构的接法。
- **与 audit_log 的区别清晰**：`audit_log` 记「**人对平台做了什么**」（operator + action，谁发布 / 谁改派）；`alert_event` 记「**系统健康事件**」（namespace/serverId 的状态异常转移），二者维度正交，不混表、不互相污染。
- **为何落库而非仅扩大内存环形**：环形缓存重启清零、容量上限丢最旧，无法回看历史；FR-89 诉求就是「历史信息流」，内存方案本质满足不了。落库是该诉求的最小可行解。
- **保留期 / 清理**：`alert_event` 是派生事件历史、可增长，类比 `metric_sample`（ADR-0023）的「保留期滚动清理」思路——**本期表结构与写入先落地，清理策略沿用既有运维设置/采样器清理范式按需补**；本期数据量（健康流转事件稀疏）远小于指标采样，暂不引清理后台，留待量级需要时再加（不提前镀金）。

## 后果

- 多一张表 + 一个 `PersistAlerter` 通道（极薄）。控制面重启后**站内信仍清零**（通道语义不变），但 `alert_event` 历史留存、可经新「事件」页回看——这正是 FR-89 要补的。
- ADR-0019「告警不落库」结论被本 ADR 在「**留痕**」维度取代：通道仍不持有真源、健康真源仍在内存；`alert_event` 是只读历史派生，不回灌健康状态、不参与任何决策。
- 将来加邮件 / IM 告警通道仍只实现 `Alerter`；加新告警类型只在发生处调 `Dispatch`——两者都自动留痕，无需再动本表 / 本通道。

## 备选方案

- **仅扩大 `InboxAlerter` 容量 / 加持久化字段**：违背站内信「进程内、重启清零」的清晰语义，且容量上限仍丢历史。被否。
- **告警历史并入 `audit_log`**：维度不同（系统事件 vs 人的操作），混表会污染审计语义与过滤。被否。
- **引入时序库 / 消息队列做告警总线**：违反「MVP 禁重型件」（ADR-0003 同源精神）。被否。
- **本期就做 ack / 去重 / 静默窗口**：超出 FR-89「历史 + 信息流」范围，属镀金。被否（需要时另写 ADR）。
