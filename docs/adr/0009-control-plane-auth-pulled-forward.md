# ADR-0009：管理面鉴权从 P2 前移

**状态**：已接受

## 背景

MVP 原定"内网可信、鉴权属 P2"（[PRD](../PRD.md) §5 与 FR-11）。但本批要做"三方插件文件覆盖兼容 + agent 执行受限重载命令"（FR-15），而控制面 admin API 当前**完全没有鉴权**（`internal/server/router.go` 的 `/admin/v1/*` 未挂任何鉴权中间件，`X-Beacon-Token` 仅防误连、非安全边界）。

在文件覆盖之前，控制面被攻破的最坏后果只是"推坏配置文本"——agent 只暴露文本、不执行任何东西。一旦给文件覆盖加上"agent 执行管理台下发的重载命令"，就等于**在一个无鉴权系统上开放了对所有子服的远程命令执行（RCE）面**。这是对抗式评审判定的安全阻断项：安全前提（鉴权）尚不存在，命令执行不能先上。

## 决策

1. **拆分原 FR-11「鉴权/配置加密」**：仅将**管理面鉴权**前移到本批；配置加密成本高、且非命令执行前置，拆为 FR-20 留后期。
2. **最小鉴权范围**（够支撑命令执行即可，不镀金）：
   - 管理台操作者认证（登录 → 令牌）；
   - 写操作（发布/回滚/改派/文件覆盖/命令下发）授权；
   - 操作者身份纳入审计，**取代前端手填的 operator 字符串**。
   - agent 侧共享 token 维持现状（仅防误连，不升级为安全边界）。
3. **命令执行与鉴权硬绑定**：鉴权未落地前，模式二（FR-15）只做"备份 + 整文件覆盖 + 由人工/插件自身重载"，**agent 执行下发命令的能力不上线**。

## 理由

- 安全前置：消除"无鉴权远程命令执行"这一致命面。
- 鉴权是多处能力的共同前置：FR-15 的命令执行、FR-18 的前端登录身份都依赖它。
- 只前移鉴权、不前移加密：加密涉及密钥管理与字段加解密，成本高且不是命令执行的前置，无谓拖慢本批。

## 后果

- **修订 [PRD](../PRD.md) §5**："MVP 内网可信（鉴权 P2）"调整为"管理面鉴权前移本批（见本 ADR）；数据面内网可信不变；配置加密仍属后期（FR-20）"。
- FR-11 收窄为"管理面鉴权"，新增 FR-20"配置加密"（P3）。
- 前端需新增登录 UI（FR-18），operator 由登录态派生而非手填。
- 本 ADR 不取代任何既有 ADR（此前无鉴权相关 ADR），仅修订 PRD 的"内网可信"表述。

## 备选方案

- **维持内网可信、不做鉴权**：则命令执行即无鉴权 RCE。否决。
- **模式二不做命令执行**：用户明确需要"执行自设重载命令"。不否决需求，改为把命令执行 gate 在鉴权之后。
- **鉴权与配置加密一起做**：加密非命令执行前置且成本高，会无谓拖慢本批；故只前移鉴权。否决。
